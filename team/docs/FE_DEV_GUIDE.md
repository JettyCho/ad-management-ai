# FE 개발 가이드: 피그마 디자인을 코드로 정확하게 옮기기

## 이 문서의 목적

이 문서는 피그마 디자인을 보고 프론트엔드 코드를 작성할 때, **빠짐없이 정확하게** 작업하기 위한 가이드다.
사람이 읽어도, AI가 읽어도, 같은 품질의 결과물을 만들 수 있도록 **원칙과 절차를 함께** 기술한다.

핵심 전제: 디자인을 코드로 옮기는 일은 "보이는 것을 베끼는 것"이 아니라, **디자인이 말하는 것과 말하지 않는 것을 모두 파악하고, 기존 코드의 맥락 위에서 올바르게 구현하는 것**이다.

---

## 1. 작업 전에 코드를 먼저 읽어라

### 왜

피그마를 먼저 보면, "이것을 어떻게 만들지?"라는 질문에서 출발하게 된다.
코드를 먼저 읽으면, "이것이 이미 어떻게 되어 있고, 무엇을 바꾸면 되지?"라는 질문에서 출발한다.

후자가 훨씬 정확하고 효율적이다. 이미 존재하는 코드는 이미 내려진 수많은 결정의 집합이다.

### 절차

**1단계: 작업 대상 파일과 주변 파일을 읽는다.**

변경해야 할 페이지 컴포넌트를 찾아 읽는다. 그 컴포넌트가 import하는 하위 컴포넌트, 훅, 타입 정의도 읽는다.

**2단계: 프로젝트의 패턴을 파악한다.**

다음 질문에 답할 수 있을 때까지 읽는다:
- 이 프로젝트의 API 호출 패턴은 무엇인가? (repo 패턴? 직접 fetch? React Query 훅?)
- 모달/오버레이는 어떻게 여는가? (useOverlay? useState? 전역 스토어?)
- 스타일링은 어떻게 하는가? (Tailwind? CSS Modules? styled-components?)
- 공용 컴포넌트(Button, Modal, Select 등)는 어디에 있고 어떤 props를 받는가?

**3단계: 토큰 체계를 파악한다.**

프로젝트의 CSS 토큰 파일(예: `buzzvil-tokens.css`, `globals.css`, `tailwind.config`)을 읽어서 다음을 파악한다:
- 색상 토큰: `--color-primary-fg-text-default`는 실제로 어떤 hex 값인가?
- 간격 토큰: `--spacing-xl`은 몇 px인가? Tailwind에서 `p-xl`로 쓸 수 있는가?
- 타이포그래피 토큰: `text-title-sm`은 어떤 font-size/weight/line-height인가?

**4단계: 재사용 가능한 기존 코드를 찾는다.**

구현해야 할 기능과 유사한 기능이 다른 페이지에 이미 있는지 검색한다. 같은 모달, 같은 API 호출, 같은 UI 패턴이 이미 구현되어 있다면 그것을 재사용한다.

## 2. 피그마에서 요소의 속성을 정밀하게 추출하라

### 왜

피그마 디자인의 스크린샷만 보고 "대충 이런 느낌"으로 구현하면, 5px 패딩 차이, 잘못된 색상 토큰, 누락된 border-radius 같은 미세한 차이가 생긴다. 하나하나는 작지만 쌓이면 전체적인 완성도가 무너진다.

### 절차

**1단계: 피그마 디자인 컨텍스트를 가져온다.**

피그마 MCP의 `get_design_context`를 호출하면 다음이 반환된다:
- 생성된 참조 코드 (React+Tailwind) — 구조와 수치를 읽는 용도
- 스크린샷 — 시각적 확인용
- 스타일 메타데이터 — 사용된 폰트, 색상, 이펙트 정보
- 컴포넌트 설명 — 디자이너가 적어둔 사용 가이드

**2단계: 바깥 컨테이너부터 안으로 읽는다.**

가장 바깥 요소부터 시작하여, 각 요소에서 다음 4가지 속성 그룹을 추출한다:

| 속성 그룹 | 확인할 것 | 확인 방법 |
|----------|----------|----------|
| **구조** | flex 방향, gap, 정렬, 크기 제약 (고정 너비? flex-1?) | 참조 코드의 className에서 flex, gap, items, justify, w-[] 확인 |
| **간격** | padding (상하좌우 각각), margin | className의 p-, px-, py-, pt-, pb-, m- 확인 |
| **시각** | 배경색, 테두리 (색/두께/radius), 그림자, 투명도 | bg-, border-, rounded-, shadow-, opacity- 확인 |
| **텍스트** | font family, size, weight, line-height, color, 정렬 | font-, text-, leading-, tracking- 확인 |

**3단계: 추출한 속성을 기존 코드와 대조한다.**

피그마에서 읽은 값과 기존 코드의 값을 **요소 단위로** 비교한다:

```
피그마: 컨테이너 너비 360px, gap 8px, 카드 패딩 px-24 py-12
코드:  컨테이너 너비 370px, gap 12px, 카드 패딩 16px (all)
→ 차이 발견: 너비 360으로, gap 8px으로, 패딩 px-xl py-md로 변경 필요
```

**4단계: 색상 값을 프로젝트 토큰에 매핑한다.**

피그마에서 색상이 나오면 다음 순서로 매핑한다:

1. 피그마 토큰 이름을 확인한다 (예: `color/primary/fg/text/default`)
2. 프로젝트 토큰 CSS에서 의미가 일치하는 토큰을 찾는다 (예: `--color-primary-fg-text-default`)
3. hex 값이 일치하는지 확인한다 (예: 둘 다 `#3b82f6`)
4. 일치하면 해당 Tailwind 클래스를 사용한다 (예: `text-primary-fg-text-default`)

**주의: 같은 hex 값에 여러 토큰이 매핑될 수 있다.** 예를 들어 `#3b82f6`이 `primary-fg-text-default`와 `status-information-stroke`에 동시에 매핑될 수 있다. 이때는 hex 값이 아니라 **피그마 토큰 이름의 의미**를 기준으로 선택한다. "주요 액션"이면 primary, "정보성 상태"면 information이다.

**5단계: 노드 이름에서 컴포넌트 역할을 읽는다.**

피그마의 `data-name` 속성은 디자이너가 이 요소를 어떤 역할로 의도했는지를 알려준다:
- `data-name="Button"` → 프로젝트의 `Button` 컴포넌트를 사용
- `data-name="Icon state"` → 프로젝트의 `IconState` 컴포넌트를 사용
- `data-name="Label"` → 의미적으로 label 역할, 프로젝트에 Label 컴포넌트가 있는지 확인

## 3. 디자인이 보여주지 않는 상태를 발견하라

### 왜

피그마는 화면의 **한 순간**을 보여준다. 코드는 **모든 순간**을 처리해야 한다.

디자인에 "담당자 미지정" 상태만 있다면, 코드는 담당자가 지정된 상태, 로딩 중 상태, 에러 상태, 확정된 후의 상태도 모두 처리해야 한다.

### 절차

**1단계: 화면에서 조건부로 바뀌는 요소를 모두 찾는다.**

"이 요소는 어떤 조건에 따라 다르게 보여야 하는가?"라는 질문을 화면의 모든 요소에 던진다.

**2단계: 각 조건부 요소에 영향을 주는 조건들을 나열한다.**

예시: 정산 담당자 카드
- 조건 A: 담당자 지정 여부 (있음 / 없음)
- 조건 B: 정산 확정 여부 (확정 / 미확정)
- 조건 C: 대행사 여부 (대행사 운영 / 아님)

**3단계: 조건들의 조합표를 만든다.**

| 담당자 | 확정 | 대행사 | 화면에 보여야 할 것 |
|--------|------|--------|-------------------|
| 없음 | 미확정 | - | "지정해 주세요" + 지정하기 버튼 |
| 있음 | 미확정 | - | 담당자 이메일 + 변경 버튼 |
| 있음 | 확정 | 아님 | 담당자 이메일 (읽기 전용) |
| 있음 | 확정 | 대행사 | 세금계산서 발행 안내 |
| 없음 | 확정 | - | ? (발생 가능한가?) |

**4단계: 디자인에 없는 조합이 발생 가능한지 검증한다.**

데이터 타입과 비즈니스 로직에서 확인한다. `billingManagerUserId`가 `number | null`이고 `statusType`이 `Confirm`일 수 있다면, "담당자 없음 + 확정" 상태는 발생 가능하다. 이 상태를 어떻게 처리할지 판단해야 한다.

**5단계: 디자인에 없는 상태는 디자인의 패턴에서 추론한다.**

피그마가 보여주는 다른 상태들의 시각적 패턴을 관찰한다:
- "액션 필요" 상태: 파란색 텍스트 + 액션 버튼 (미지정 상태에서 사용)
- "정보 표시" 상태: 회색 라벨 + 검정 값 (광고주 카드에서 사용)

이 패턴에서 추론하면, "담당자 지정됨" 상태는 "정보 표시" 패턴(회색 라벨 + 이메일)을 따르는 것이 일관성 있다.

## 4. 피그마의 코드가 아니라 의도를 읽어라

### 왜

피그마 MCP가 생성하는 코드는 Tailwind으로 된 참조 코드다. 이 코드를 그대로 쓰면:
- 프로젝트의 컴포넌트 대신 raw div를 쓰게 된다
- 프로젝트의 토큰 대신 하드코딩된 값을 쓰게 된다
- 프로젝트의 패턴과 다른 구조가 된다

### 판단 기준

피그마 출력의 각 부분을 다음 기준으로 판단한다:

| 피그마가 주는 것 | 어떻게 사용하는가 |
|----------------|-----------------|
| 레이아웃 구조 (flex, gap, padding 등) | **수치**만 참고하고, 프로젝트의 토큰으로 변환 |
| 색상 값 (#hex) | 프로젝트 토큰 CSS에서 매칭되는 시맨틱 토큰을 찾아서 사용 |
| 컴포넌트 이름 (data-name="Button") | 프로젝트의 동일 역할 컴포넌트를 사용 |
| 인라인 스타일, 절대 위치 | **무시**. 프로젝트의 레이아웃 패턴으로 대체 |
| 이미지 에셋 URL | 프로젝트에 이미 있는 아이콘/이미지면 기존 것 사용 |

## 5. 보이는 것과 보이지 않는 것을 분리하여 구현하라

### 왜

UI(보이는 층)와 로직(보이지 않는 층)을 동시에 구현하면, 둘 다 중간 수준으로 끝나기 쉽다. 분리하면 각각에 집중할 수 있다.

### 절차

**보이는 층 먼저:**
- 레이아웃 구조를 기존 코드에 반영한다 (너비, 패딩, gap 등 수치 조정)
- 조건부 렌더링 분기를 만든다 (3장에서 만든 상태 조합표 기준)
- 각 분기에서 올바른 텍스트, 색상, 컴포넌트를 사용한다

**보이지 않는 층 다음:**
- 필요한 데이터를 가져오는 훅을 추가한다
- 사용자 인터랙션 핸들러를 구현한다
- 권한/역할에 따른 분기 로직을 구현한다
- 데이터 변경 후 캐시 무효화를 처리한다

보이지 않는 층에서 특히 주의할 것:

**캐시 무효화:** 화면 A에서 데이터를 변경했을 때, 같은 데이터를 표시하는 화면 B의 쿼리도 무효화되어야 한다. 기존 mutation 훅이 무효화하는 쿼리 목록을 확인하고, 새 맥락에서 추가로 필요한 무효화가 있는지 판단한다.

**로딩 상태 안전성:** `query.data`가 undefined일 때 `query.data.id`에 접근하면 런타임 에러가 발생한다. 로딩 중에는 안전한 기본값(비활성화, 숨김, 스켈레톤)을 반환해야 한다.

## 6. 기존 코드에 없는 것만 만들어라

### 왜

이미 있는 것을 다시 만들면 두 가지 문제가 생긴다:
1. 불필요한 코드가 늘어난다
2. 기존 것과 미세하게 다른 동작이 생겨서 일관성이 깨진다

### 판단 기준

구현해야 할 기능을 분해했을 때:

| 이미 있는 것 | 새로 만들어야 하는 것 |
|------------|-------------------|
| 모달 컴포넌트 | 모달을 여는 트리거 UI |
| API 호출 훅 (mutation/query) | 새 맥락에서의 onSuccess/onError 콜백 |
| 공용 UI 컴포넌트 (Button, Select 등) | 상태에 따른 조건부 렌더링 로직 |
| 데이터 타입 정의 | (대부분 이미 있다) |

재사용할 때 반드시 확인할 것: 기존 훅의 `onSuccess`에서 무효화하는 쿼리 목록이 새 맥락에서도 충분한가? 부족하다면 새 맥락의 콜백에서 추가 무효화를 처리한다.

## 7. 다각도로 검증하라

### 왜

하나의 시각에서만 검증하면, 다른 시각에서의 결함을 놓친다.

### 절차: 세 가지 축으로 검증

**시간 축:**
페이지 진입 → 로딩 중 → 데이터 도착 → 사용자 액션 → 액션 성공/실패. 각 시점에서 화면이 올바른지 확인한다.

**권한 축:**
같은 화면을 다른 역할의 사용자가 보면 어떻게 되는가? 정산 담당자 본인 vs 다른 멤버 vs 미지정 상태. 각 경우에 버튼 활성화/비활성화, 표시/숨김이 올바른지 확인한다.

**데이터 흐름 축:**
데이터를 변경하면 화면의 모든 관련 부분이 갱신되는가? 변경된 데이터가 다른 쿼리에 캐시되어 있다면, 그 쿼리도 무효화되는가?

### 절차: 브라우저에서 실제 확인

TypeScript 컴파일이 통과해도 반드시 브라우저에서 확인한다:
- 레이아웃이 피그마와 일치하는가? (간격, 정렬, 텍스트 줄바꿈)
- 색상이 정확한가? (토큰이 의도한 값과 다른 hex로 매핑되어 있을 수 있다)
- 인터랙션이 정상 동작하는가? (버튼 클릭, 모달 열기/닫기)
- 다른 상태에서도 화면이 깨지지 않는가?

## 8. 작업 순서 요약

1. **코드를 읽는다.** 변경 대상 파일, 관련 컴포넌트, 타입, API 훅, 토큰 체계를 파악한다.
2. **피그마 디자인 컨텍스트를 가져온다.** `get_design_context`로 참조 코드, 스크린샷, 메타데이터를 얻는다.
3. **피그마 속성을 정밀하게 추출한다.** 바깥에서 안으로, 각 요소의 구조/간격/시각/텍스트 속성을 읽고, 기존 코드와 대조하여 차이를 찾는다.
4. **색상과 간격을 프로젝트 토큰에 매핑한다.** hex 값이 아니라 피그마 토큰 이름의 의미를 기준으로 프로젝트 토큰을 선택한다.
5. **모든 상태를 나열한다.** 조건부 요소의 조건 조합표를 만들고, 디자인에 없는 상태의 처리 방법을 결정한다.
6. **보이는 층을 구현한다.** 레이아웃, 스타일, 조건부 렌더링을 매핑된 토큰으로 구현한다.
7. **보이지 않는 층을 구현한다.** 데이터 페칭, 상태 관리, 권한 검사, 캐시 무효화를 구현한다.
8. **세 축으로 검증한다.** 시간/권한/데이터 흐름 축에서 로직을 검증한다.
9. **브라우저에서 확인한다.** 피그마 스크린샷과 실제 화면을 비교한다.

이 순서에서:
- **5번이 가장 중요하다.** 상태를 빠짐없이 나열하는 것이 나머지 모든 작업의 품질을 결정한다.
- **3~4번이 가장 놓치기 쉽다.** 속성을 대충 읽으면 "비슷하지만 정확하지 않은" 결과물이 나온다.
- **7번의 캐시 무효화가 가장 발견하기 어려운 버그를 만든다.** 화면에 보이지 않는 데이터 의존성을 놓치면, 변경이 반영되지 않는 stale UI가 된다.
