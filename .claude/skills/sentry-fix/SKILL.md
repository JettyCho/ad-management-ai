# Sentry 이슈 분석 및 수정

## 입력값 파싱

전달받은 인자: `$ARGUMENTS`

사용자 입력에서 Sentry 이슈 정보를 추출한다:

- **Sentry URL**: `sentry.io` 가 포함된 URL → 그대로 `get_issue_details`의 `issueUrl`로 사용
- **이슈 ID**: `BUZZAD-XXXX` 형식 → `organizationSlug: buzzvilcom`, `issueId`로 사용
- **자연어 설명**: 에러 메시지, 스택트레이스 키워드 등 → `search_issues`로 검색
- **슬랙 공유 내용**: Sentry 봇이 보낸 메시지에서 URL이나 이슈 ID를 추출

## 환경 설정

adserver 코드베이스 경로는 `.env` 파일의 `ADSERVER_PATH`에서 읽는다.
`.env` 파일이 없거나 `ADSERVER_PATH`가 설정되지 않았으면 사용자에게 경로를 물어본다.

---

## 1단계: Sentry 이슈 상세 수집

`get_issue_details`로 아래 정보를 수집한다:

- 에러 메시지 및 에러 타입
- 전체 스택트레이스 (first-party 코드 위주로 분석)
- 로컬 변수 값
- 발생 빈도 (occurrences), 영향 사용자 수
- 최초/최근 발생 시각
- 환경 (prod/staging), 릴리즈 버전
- HTTP 요청 정보 (엔드포인트, 메서드)
- 태그 및 추가 컨텍스트

수집 후 사용자에게 이슈 요약을 먼저 보여준다:
- 어떤 에러인지
- 얼마나 자주 발생하는지
- 어디서 발생하는지 (엔드포인트, 파일, 라인)

## 2단계: 코드베이스 심층 조사 (가장 중요한 단계)

**⚠️ 이 단계가 전체 작업의 품질을 결정한다. 절대 서두르지 않는다.**

`ADSERVER_PATH`의 코드베이스에서 에러 지점을 중심으로 조사한다.

### 2-1. 에러 지점 직접 확인

스택트레이스의 first-party 코드 프레임을 위에서 아래로 모두 읽는다:
- 에러가 발생한 정확한 라인과 주변 코드
- 해당 함수의 전체 시그니처와 파라미터

### 2-2. 호출 체인 전체 추적

에러 지점만 보지 않는다. **호출 체인을 거슬러 올라가며 전체 흐름을 파악한다**:
- 에러를 유발하는 값(예: None인 변수)이 **어디서 생성되는지**
- 그 값이 **어떤 조건에서 None/에러 상태가 되는지**
- 중간 호출 함수들이 **에러 상태를 어떻게 전파하는지**

### 2-3. 의도 분석 (핵심)

단순히 "여기서 None이니까 체크하면 됨"이 아니다. 반드시 아래를 검토한다:

1. **이 에러가 의도된 것인가?**
   - 누군가 일부러 에러를 발생시켜 문제를 감지하려 한 건 아닌지
   - 에러 핸들링 미들웨어나 로깅이 이 에러를 잡아서 모니터링하고 있는 건 아닌지

2. **같은 변수에 대한 기존 방어 코드가 있는가?**
   - 다른 곳에서 이미 None 체크를 하고 있다면, 이 에러는 방어 코드 누락
   - 아무 곳에서도 체크하지 않는다면, 원래 None이 올 수 없다고 가정한 설계일 수 있음

3. **수정 시 사일런트 페일이 생기지 않는가?**
   - None 체크 후 단순 스킵이 올바른 동작인지 논리적으로 검증
   - 스킵하면 안 되는 중요한 로직이 빠지진 않는지
   - 에러 응답이 클라이언트에 올바르게 전달되는지

4. **근본 원인 vs 증상 수정**
   - None 체크 추가가 근본 해결인지, 아니면 상위에서 흐름 자체를 바꿔야 하는지
   - 예: 애초에 None을 반환하지 않도록 상위 함수를 수정하는 게 맞을 수도 있음

### 2-4. 영향 범위 파악

- 수정 대상 코드를 사용하는 다른 호출자가 있는지
- 동일한 패턴의 에러가 다른 곳에도 존재하는지 (같은 실수의 반복)

## 3단계: 수정 방안 도출 및 사용자 확인

조사 결과를 사용자에게 보고하고, 수정 방안을 제시한다:

### 보고서 형식

```
### Sentry 이슈 분석 결과

| 항목 | 내용 |
|------|------|
| **이슈** | [이슈 ID](Sentry URL) — 에러 메시지 |
| **발생 빈도** | N회 (기간) |
| **엔드포인트** | HTTP 메서드 + 경로 |
| **에러 위치** | 파일:라인 (함수명) |

### 근본 원인
(호출 체인 분석 결과를 포함한 상세 설명)

### 의도 분석
(이 에러가 왜 발생하는지, 의도된 것인지, 방어 코드 누락인지)

### 수정 방안
(구체적인 변경 내용과 그 이유)

### 사일런트 페일 검증
(수정이 논리적으로 올바른 이유)
```

**사용자가 수정 방안을 확인하고 동의한 후에만 4단계로 진행한다.**

## 4단계: 코드 수정 및 PR 생성

### 4-1. 브랜치 생성

```
cd $ADSERVER_PATH
git checkout master && git pull origin master
git checkout -b fix/sentry-{이슈ID}-{간결한-설명}
```

### 4-2. 코드 수정

- 최소한의 변경으로 문제를 해결한다
- 불필요한 리팩토링이나 코드 정리를 하지 않는다
- 수정 내용이 기존 동작을 깨뜨리지 않는지 확인한다

### 4-3. 커밋

커밋 메시지는 아래 형식을 따른다:

```
fix: {한 줄 요약}

{상세 설명 - 왜 이 에러가 발생했는지, 어떻게 수정했는지}

Fixes {SENTRY_ISSUE_ID}

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

### 4-4. Push 및 PR 생성

`gh` CLI를 사용하여 PR을 생성한다:

```bash
source .env
git push origin {브랜치명}
GH_TOKEN=$GITHUB_TOKEN gh pr create --repo Buzzvil/adserver \
  --base master \
  --head {브랜치명} \
  --title "fix: {한 줄 요약}" \
  --body "$(cat <<'EOF'
PR 본문 (아래 형식 참고)
EOF
)"
```

PR 본문에는 반드시 아래를 포함한다:

1. **Summary**: Sentry 이슈 링크, 발생 빈도, 에러 요약
2. **Root Cause**: 호출 체인 분석을 포함한 근본 원인 (단순 증상이 아닌 왜 이 상황이 발생하는지)
3. **의도 분석 근거**: 방어 코드 누락의 증거 (예: 다른 곳에서 이미 None 체크를 하고 있었다는 사실)
4. **Changes**: 구체적으로 무엇을 어떻게 바꿨는지
5. **사일런트 페일 검증**: 수정이 올바른 동작인 이유
6. **Test plan**: 검증 체크리스트

---

## 중요 원칙

1. **급하게 고치지 않는다** — 다각도 검토를 반복하는 것이 좋은 결과를 만든다
2. **의도를 먼저 파악한다** — 코드 작성자가 왜 이렇게 짰는지 이해한 후 수정한다
3. **사일런트 페일을 경계한다** — 에러를 숨기는 것과 올바르게 처리하는 것은 다르다
4. **최소 변경 원칙** — 문제 해결에 필요한 최소한의 변경만 한다
5. **증거 기반** — 추측이 아닌 코드에서 발견한 증거를 기반으로 판단한다

보고서는 한국어로 작성한다. 기술 용어나 고유명사는 원문 그대로 사용한다.
